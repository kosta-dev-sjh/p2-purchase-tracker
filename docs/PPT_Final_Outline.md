**제목**: SpendTrack
**부제**: 사진 한 장으로 끝내는 소비 기록 — 온라인 쇼핑 시대의 가계부
**부속 정보**: 발표일 / 팀명 / 발표자명 / 데모 URL (Firebase Hosting)
**시각 자료**:

- 우측: 표지 메시지
- 배경: 흰색 + 인디고 그라데이션 strip 한 줄

**발표자 한 줄**: "오늘은 카드 명세서만 봐서는 알 수 없던 '진짜 소비'를 어떻게 다시 손에 쥐는지 보여드립니다."

## 슬라이드 1 — 왜 SpendTrack 인가 (필요성)

**제목**: 온라인은 쉽고 빠르지만, 기록은 흩어진다
**핵심 문장**: 카드사·쇼핑몰·앱마다 따로 남는 구매 흔적이, 한 명의 소비자 안에서 한 번도 합쳐지지 않습니다.

**근거 (조사 후 출처 포함 — 발표자 노트로 보강)**
- 통계청 「온라인쇼핑동향」: 국내 온라인쇼핑 거래액 매년 두 자릿수 성장 — 2024년 200조 원 돌파.
- 한국은행 「지급결제 보고서」: 카드결제 중 비대면 비중이 코로나 이후 50%대 진입, 회복기에도 떨어지지 않음.
- 모바일 결제 보편화 → 카드 명세서에 "쇼핑몰 이름과 금액"만 남고, "무엇을 샀는지"는 사라짐.

**시각 자료**
- 막대 그래프: "2019 → 2024 온라인 거래액 추이" (출처: 통계청, 발표자 노트 필수)
- 우측: 카드 명세서 캡처 모형 — `쿠팡 139,000원` 한 줄에 박스
- 박스 옆 콜아웃: "뭘 샀더라?"

**디자이너 노트**: 통계는 한 개만, 시각적으로 가장 명확한 것 한 장. 숫자 폭격 금지.

## 슬라이드 2 — 누가 쓰는가 (타겟 사용자)

**제목**: 기록은 하고 싶은데, 손으로 일일이 치는 건 싫은 사람들
**핵심 문장**: 온라인 쇼핑에 익숙하지만, 가계부 앱 입력의 마찰 때문에 매번 중도 포기한 경험이 있는 사용자가 1차 타겟입니다.

**페르소나 3종 (강사 피드백 반영)**

| 페르소나 | 한 줄 정의 | 핵심 페인 |
| --- | --- | --- |
| 김지원 (28, 직장인) | 쿠팡·네이버를 일주일에 3회 이상 결제 | 카드 명세서를 봐도 뭘 샀는지 기억 안 남 |
| 박혜린 (35, 워킹맘) | 생활비·교육비 함께 관리 | 환불·취소·할부가 섞여 실제 지출 추적 안 됨 |
| 이수민 (42, 1인 가구) | 넷플릭스·정기배송 등 구독 다수 | 반복결제 합계가 한 화면에서 안 보임 |

**근거 (조사 후 출처 포함)**
- 통계청 「가계동향조사」: 30대 1인 가구의 비대면 소비 지출 비중이 60% 상회.
- 모바일 가계부 앱 사용자 이탈률 — 수동 입력 마찰이 1순위 이탈 사유 (앱 분석 리포트들).

**시각 자료**: 페르소나 3장 카드 (사진 자리 + 페인 1줄 + 사용 패턴 1줄)

## 슬라이드 3 — 문제 정의

**제목**: 우리가 푸는 문제 3가지
**핵심 문장**: 자동화가 막힌 자리에서, 사용자는 결국 "직접 입력"이라는 마찰에 부딪힙니다.

**3개 문제 블록**

1. **카드 내역 ≠ 실제 소비**
   - "쿠팡 139,000원" 한 줄로는 무엇을 샀는지, 환불 됐는지, 일시불인지 할부인지 알 수 없음.
2. **자동 가져오기가 막혀 있음**
   - 국내 쇼핑몰 API 는 판매자용만 존재 — 구매자 본인의 주문 내역을 외부로 내보내는 공식 경로가 없음.
   - 사이트 자동 크롤링은 약관 위반 위험.
   - 결국 남는 합법·현실적 경로: **사용자가 직접 캡처해서 올리기**.
3. **반복결제는 한 화면에서 안 보임**
   - 카드사 앱마다 정기결제 표시가 다르고, 통신비·구독료·할부가 분산.
   - 해지 누락 → "썼는지도 몰랐던 결제" 발생.

**시각 자료**
- 좌: 카드 명세서 mock + "?" 콜아웃
- 중: 쇼핑몰 API 표 (쿠팡/네이버/무신사/카카오페이/G마켓 — 전부 "구매자용 외부 노출 X")
- 우: 캘린더 위에 흩어진 반복결제 아이콘들

## 슬라이드 4 — 기존 서비스와의 차별점

**제목**: 단순 가계부가 아닙니다 — 쇼핑 단위까지 들어갑니다
**핵심 문장**: 카드사 앱은 "결제 단위"에서 끝나고, 가계부 앱은 "수기 입력"에서 멈춥니다. SpendTrack 은 그 사이를 OCR + AI 보정으로 메웁니다.

**비교표**

| 비교 항목 | 카드사 앱 | 일반 가계부 앱 | SpendTrack |
| --- | --- | --- | --- |
| 입력 방식 | 자동(카드 결제만) | 수동 입력 | 사진 OCR + CSV/XLSX + 수동 |
| 무엇을 샀는지 | ❌ | △ (수동 작성 시) | ✅ 상품 단위 추출 |
| 환불·취소 추적 | △ | ❌ | ✅ 상태 태그 + KPI 분리 |
| 반복결제 자동 분류 | ❌ | △ | ✅ 4종 분류(구독/공과금/할부/빈출) |
| 다중 플랫폼 통합 | 카드 단일 | 수동 의존 | 카드 + 쿠팡 + 네이버 통합 |
| AI 보정 | ❌ | ❌ | ✅ Gemini 2.5 Flash (선택적) |

**시각 자료**: 표는 좌–우로 SpendTrack 컬럼만 인디고 강조

## 슬라이드 5 — 핵심 기능 5개

**제목**: 5가지 기능, 한 화면 안에서
**핵심 문장**: 입력의 다양성 → 정제의 일관성 → 분석의 통합성. 세 단계가 한 흐름 안에서 끊기지 않습니다.

**5 카드**

1. **이미지 OCR (쿠팡·네이버)**
   - tesseract.js 브라우저 1차 OCR (kor+eng) → 플랫폼별 파서 → 품질 평가
   - 쿠팡 23장 ground-truth harness 통과 기준 회귀 대응 모드
2. **CSV / XLSX 카드 내역**
   - 표준 카드사 헤더(이용일/가맹점명/이용금액/할부개월/할부회차) 자동 매핑
   - 헤더 매칭 0건 시 AI fallback 1회 호출 (시트 단위)
3. **AI fallback (선택적)**
   - Gemini 2.5 Flash Vision — bad 카드 있는 이미지에만 발동
   - 실제 값이 바뀐 카드만 `aiApplied` 표시 (거짓 강조 금지)
4. **반복결제 자동 탐지**
   - 4종 분류: subscription / utility / installment / frequent
   - 2개월+ 동일 금액(±15%) 자동 인식, 가맹점 개념 사전 + 할부 메타 활용
5. **소비 인사이트**
   - 월별 KPI · 플랫폼/카테고리 도넛/막대 · 요일 패턴 · 반복결제 합계
   - AI 1~2문장 요약 (월별 캐시, 데이터 변동 시에만 재호출)

**시각 자료**: 5개 카드 그리드 (각 카드: 아이콘 + 1줄 정의 + 작은 코드/숫자 한 조각)

## 슬라이드 6 — 유저 플로우

**제목**: 로그인부터 분석까지 5단계
**핵심 문장**: 사용자는 "사진을 올린다" 만 하면 됩니다. 나머지는 OCR · 품질 평가 · AI 보정 · 중복 검사가 알아서 굴러갑니다.

**플로우 다이어그램**

```
① 로그인 (Email/PW or Google OAuth)
        ↓
② 입력 선택 (OCR · 수동 · 카드 파일)
        ↓
③ 정제
   - 이미지: tesseract OCR → 플랫폼 파서 → 품질 평가 → (선택) Gemini Vision 보정
   - CSV/XLSX: 헤더 자동 매핑 → (선택) AI 행 분류
   - 수동: 직접 입력 + 후보 거래 자동 매칭
        ↓
④ 검수 / 저장
   - OcrEdit: 카드별 편집 + 합계 점검 + 중복 처리(skip/merge/new)
   - 거래 매칭 모달: 강제 머지 없음, 사용자 선택
        ↓
⑤ 분석
   - Home: KPI · 도넛 · 추이 · AI 인사이트
   - Analysis: 카테고리/요일/반복결제 심층
   - Subscriptions: 정기결제 전용
```

**시각 자료**: 5박스 가로 흐름. ②③④ 박스에 작은 화면 캡처 미니어처. ③ 박스 안에서만 분기(OCR/CSV/Manual) 표시.

## 슬라이드 8 — SWOT 분석

**제목**: 현재 위치 진단
**핵심 문장**: 2년 전 기획 SWOT 를 현재 빌드 기준으로 다시 그렸습니다.

**4 사분면**

| 강점 (Strength) | 약점 (Weakness) |
| --- | --- |
| 쇼핑 단위 OCR 추출 (쿠팡/네이버) | 신규 플랫폼(테무 등) 인식률 미보장 |
| AI 비용 통제 게이트(`pickBadProducts`) | OCR 정확도가 사용자 캡처 품질에 의존 |
| Firestore 실시간 동기 | Firestore Rules 정식 적용 전 (막바지 hardening) |
| Functions Secret 으로 API 키 분리 | 모바일 앱 미지원 (현재 PWA 수준) |
| 카드 CSV/XLSX 표준 헤더 + AI fallback | 영수증 OCR 미지원 |

| 기회 (Opportunity) | 위협 (Threat) |
| --- | --- |
| 비대면 결제 비중 50%+ 고착 | 쇼핑몰 UI 변경 시 OCR 파서 회귀 |
| 가계부 앱 시장의 "수동 입력 이탈" 미해결 | Gemini 가격 정책 변동 가능 |
| 구독 경제 확산 → 반복결제 관리 수요 | 약관/스크래핑 규제 변화 |
| Functions 기반 점진적 SaaS 확장 가능 | 사용자 캡처 화면 표준이 없음 |

**시각 자료**: 2x2 그리드, 인디고/그린/앰버/레드 4색 토큰 적용. 각 셀당 5개 이내.

## 슬라이드 9 — 운영 비용 & 사업성

**제목**: 운영비는 매우 낮습니다 — 비용의 거의 전부가 "선택적 AI 호출"
**핵심 문장**: 정적 프론트 + Firebase 무료 티어 + Gemini 후불제(2.5 Flash) 조합으로, 실 사용 패턴 기준 1인 월 운영비는 수십 원 안쪽입니다.

**비용 구조 (현재 코드 기준)**

| 항목 | 단가 / 무료 한도 | 실제 사용 패턴 |
| --- | --- | --- |
| Firebase Hosting | 10GB 대역/월 무료 | 정적 자산 ~수 MB, 무료 안쪽 |
| Cloud Firestore | 50K read/day · 20K write/day · 1GB 무료 | 1인 사용 패턴은 무료 한도 안쪽 |
| Cloud Functions | 2M invocation/월 무료 | 호출 대부분 `geminiProxy` (월 수십 회) |
| Gemini 2.5 Flash (paid) | 입력 ≈ $0.30/1M 토큰, 출력 ≈ $2.50/1M 토큰 | 호출당 약 $0.001~$0.003 |

## 슬라이드 10 — 보안 현황 & 향후 hardening (신규)

# 부록 C — 슬라이드 모션 / 애니메이션 기획안

**전체 원칙**

- 톤은 "기업 제품 발표". 학생 발표 톤의 통통 튀는 모션, 회전, 줌, 사선 슬라이드는 금지.
- 기본 트랜지션: 슬라이드 간은 **fade(0.3s)** 한 종류만.
- 객체 단위 모션: **fade-in + 8~12px slide-up** 조합만 사용. 가속도는 ease-out, 0.25~0.4s.
- 시퀀스 간 간격(stagger): 0.08~0.12s. 너무 길면 발표 흐름이 끊김.
- 표지(0)·마무리(12) 만 살짝 정성을 들이고, 본문 슬라이드는 "정보가 한 박자씩 도착하는 느낌" 정도.
- 한 슬라이드당 모션 단계는 **5단계 이내**. 발표자가 클릭 한두 번으로 끝낼 수 있어야 함.

# 부록 E — 발표 말투 가이드

이 문서의 슬라이드 본문은 **읽기 좋은 문서 톤**으로 작성되어 있습니다. 그대로 읽으면 어색합니다. 발표 시에는 아래 원칙으로 다시 풀어서 말씀하세요.

## E-1. 핵심 원칙 5가지

1. **슬라이드는 보조 자료, 입에서 나오는 말이 본체**
   슬라이드 문장을 그대로 읽지 않습니다. 슬라이드는 키워드만 노출하고, 발표자가 그 키워드 사이를 자연스러운 말로 잇습니다.
2. **모든 문장은 "~습니다 / ~합니다" 정중체**
   기업 제품 발표 톤. "~예요 / ~죠 / ~잖아요" 같은 친근체는 발표 흐름에서 가볍게 보입니다. 단, 부자연스러울 정도로 딱딱하면 안 됩니다.
3. **한 호흡 = 한 문장 = 12~18자**
   긴 문장은 듣는 사람이 따라오기 힘듭니다. "주어 + 핵심 동사" 단위로 끊어 말합니다.
4. **숫자·고유명사는 또박또박**
   "200조 원", "₩15에서 ₩50", "Gemini 2.5 Flash" 같은 핵심 숫자/이름은 한 박자 쉬고 천천히. 청중이 받아쓸 시간을 줍니다.
5. **장면 전환에는 짧은 연결어**
   "그런데", "그래서", "여기서 한 가지", "이어서", "다음으로" — 이 다섯 개만 돌려 써도 흐름이 끊기지 않습니다. "음", "어" 같은 군더더기는 의식적으로 제거.

## E-3. 슬라이드별 발표 멘트 예시

각 슬라이드를 약 30~50초 안에 마치는 분량으로 작성했습니다. 발표자가 자기 말투로 다듬어 사용하시면 됩니다.

### 슬라이드 0 — 표지 (15~20초)

> "안녕하세요. 오늘 보여드릴 서비스는 SpendTrack 입니다.
> 한 줄로 요약하면, **사진 한 장으로 끝내는 소비 기록** 입니다.
> 온라인 쇼핑에 익숙한 분들을 위한, 카드 명세서가 채우지 못하는 영역을 메우는 가계부 서비스입니다."

### 슬라이드 1 — 왜 SpendTrack 인가 (40~50초)

> "통계청 자료 기준으로, 국내 온라인쇼핑 거래액은 2024년에 200조 원을 돌파했습니다.
> 매년 두 자릿수로 성장 중입니다.
> 그런데 카드 명세서를 보면 이런 줄이 남습니다.
> *(우측 카드 명세서 가리키며)*
> '쿠팡 139,000원' — **무엇을 샀는지** 알 수가 없습니다.
> 결제는 끝났는데 기억은 사라집니다.
> 저희가 풀려고 한 문제가 바로 이겁니다."

### 슬라이드 2 — 누가 쓰는가 (35~45초)

> "타겟은 **온라인 쇼핑에 익숙하지만, 가계부 입력의 마찰 때문에 매번 포기한 경험이 있는 사용자**입니다.
> 세 명의 페르소나로 나눠봤습니다.
> 김지원 님은 일주일에 세 번 이상 결제하시는 직장인. 카드 명세서를 봐도 뭘 샀는지 기억이 안 납니다.
> 박혜린 님은 워킹맘. 환불·취소·할부가 섞여서 실제 지출이 잘 안 보입니다.
> 이수민 님은 1인 가구. 구독료가 한 화면에서 합쳐서 안 보입니다.
> 공통점은 하나입니다. **기록은 하고 싶은데 손으로 일일이 치는 건 싫다.**"

### 슬라이드 3 — 문제 정의 (35~40초)

> "문제를 세 가지로 정리했습니다.
> 첫째, 카드 내역과 실제 소비가 다릅니다. 한 줄짜리 결제 정보로는 무엇을 샀는지 알 수 없습니다.
> 둘째, 자동 가져오기가 막혀 있습니다. 국내 쇼핑몰 API 는 판매자용만 있고, 구매자용 외부 노출은 어디에도 없습니다. 크롤링은 약관 위반 위험이 있고요.
> 셋째, 반복결제가 한 화면에 안 보입니다. 카드사 앱마다 표시 방식이 다르고, 통신비·구독료·할부가 분산되어 있습니다.
> 자동화가 막힌 자리에서 **합법적이고 현실적인 유일한 해법은 사용자가 캡처한 화면을 정제하는 것** 입니다."

### 슬라이드 4 — 차별점 (30초)

> "기존 서비스와 비교하면 차이가 분명해집니다.
> 카드사 앱은 결제 단위에서 끝납니다. 가계부 앱은 수기 입력에서 멈춥니다.
> 저희는 그 사이를 OCR 과 AI 보정으로 메웠습니다.
> 단순한 가계부가 아니라, **상품 단위까지 들어가는 소비 추적기** 입니다."

### 슬라이드 5 — 핵심 기능 5개 (60초, 가장 길게)

> "기능은 다섯 가지입니다.
> 첫째, **이미지 OCR**. 쿠팡과 네이버 주문 캡처를 인식합니다. 브라우저에서 1차 분석을 하고, 쿠팡 기준으로는 23장 샘플 ground-truth 테스트를 통과한 상태입니다.
> 둘째, **카드 CSV / XLSX 일괄 업로드**. 표준 카드사 헤더를 자동으로 매핑하고, 매핑이 0건일 때만 AI 가 한 번 더 봅니다.
> 셋째, **AI 보정 — 단, 선택적으로만**. 이미지 품질이 나쁜 카드에서만 Gemini 2.5 Flash 가 발동합니다. 멀쩡한 사진에는 호출하지 않아 비용을 통제합니다.
> 넷째, **반복결제 자동 분류**. 구독·공과금·할부·빈출 네 종류로 자동으로 나눕니다.
> 다섯째, **소비 인사이트**. 월별 KPI 와 함께 한두 문장의 AI 요약을 제공합니다.
> 다섯 가지 기능이 한 화면에서 끊김 없이 흐릅니다."

### 슬라이드 6 — 유저 플로우 (30초)

> "사용자 입장에서는 다섯 단계입니다.
> 로그인하고, 입력 방식을 고르고, 정제하고, 검수해서 저장하고, 분석합니다.
> 핵심은 세 번째 정제 단계입니다.
> 이미지면 OCR 과 AI 보정이, 카드 파일이면 자동 매핑과 AI 분류가, 수동 입력이면 후보 거래 자동 매칭이 알아서 굴러갑니다.
> 사용자는 사실상 **사진 한 장 올리는 행위만** 하면 됩니다."

### 슬라이드 7 — 기술 스택 (35초)

> "기술 스택은 신기술 자랑이 아니라, **검증된 표준 조합**을 원칙으로 했습니다.
> 프론트는 React 19 와 Vite. 상태는 Zustand 다섯 스토어로 분리했습니다.
> OCR 은 tesseract.js, AI 는 Google Gemini 2.5 Flash 의 비전 모델입니다.
> 인증과 데이터, 그리고 서버는 Firebase 한 묶음으로 갔습니다.
> *(우측 다이어그램 가리키며)*
> AI 호출은 무조건 Functions 를 거칩니다. 키는 프론트 번들에 절대 노출되지 않습니다."

### 슬라이드 8 — SWOT (40초)

> "현재 위치를 SWOT 으로 정리했습니다.
> 강점은, 쇼핑 단위 OCR 추출과 AI 비용 통제 게이트, 그리고 키 분리 완료입니다.
> 약점은, 신규 플랫폼 인식률 미보장, 그리고 Firestore Rules 정식 적용이 막바지 작업으로 남아 있다는 점입니다.
> 기회는, 비대면 결제가 50% 위에서 고착되었고, 구독 경제가 계속 확산 중이라는 시장 흐름입니다.
> 위협은, 쇼핑몰 UI 변경 시 OCR 파서 회귀, 그리고 Gemini 가격 정책 변동입니다.
> 위협은 인지하고 있고, 회귀 대응 모드로 동결했습니다."

### 슬라이드 9 — 운영 비용 (45~50초)

> "여기가 저희가 자랑할 만한 부분입니다.
> 운영 구조는 정적 프론트, Firebase 무료 티어, 그리고 Gemini 후불제 — 세 가지로 끝납니다.
> *(좌측 비용표 가리키며)*
> Hosting, Firestore, Functions 모두 무료 한도 안쪽입니다.
> 비용의 거의 전부가 **Gemini 호출 — 그것도 선택적으로 발동되는 호출**입니다.
> 실제 사용 패턴 기준으로 1인 월 운영비를 추정하면…
> *(₩15~₩50 카드 가리키며)*
> 15원에서 50원입니다. 천 원이 아니라 50원입니다.
> 한계 비용이 낮기 때문에, 구독 모델로 전환하면 단가의 90% 이상이 마진으로 남습니다."

### 슬라이드 10 — 보안 현황 (40초)

> "보안은 단계로 쌓고 있습니다. 한꺼번에 처리하지 않고 세 단계로 나눴습니다.
> *(현재 적용 박스 가리키며)*
> MVP 기준 적용된 항목입니다. Firebase Auth, 키 분리, 계정 삭제 7일 grace, 닉네임 쿨다운, 비밀번호 정책 통일까지 들어가 있습니다.
> *(막바지 박스)*
> 출시 전에 묶어서 처리할 항목입니다. Firestore Rules 정식 적용, 이메일 변경 실 Auth 연동, 이메일 검증 강제 흐름.
> *(선택 박스)*
> 사용자 규모와 리스크를 보고 도입을 결정할 항목입니다. App Check, MFA, rate limit.
> **기능 우선 MVP 단계**라는 점을 명확히 하고, 보안 강화는 별도 이슈로 트래킹하고 있습니다."

### 슬라이드 11 — 향후 계획 (35초)

> "다음 단계는 네 축으로 확장합니다.
> OCR 은 네이버 인식률을 개선하고, 신규 플랫폼과 영수증 OCR 까지 넓힙니다.
> 카드 내역은 PDF 명세서 대응을 추가합니다.
> 반복결제는 보험사와 공과금까지 끌어들여 4분류를 8분류로 늘립니다.
> 마지막으로, 플랫폼 제약을 점진적으로 완화합니다.
> 원칙은 하나입니다. **지금 잘 하는 것을 흔들지 않고 확장한다.**"

### 슬라이드 12 — 마무리 (20초)

> "다시 한 줄로 정리하면, 이렇습니다.
> *(큰 카피 가리키며)*
> 카드 내역만 봐서는 모르는 문제를, 사진 한 장으로 풉니다.
> 입력 다양성, 정제 일관성, 분석 통합성 세 가지가 한 흐름 안에 들어가 있는 가계부입니다.
> 여기까지가 SpendTrack 입니다.
> 질문 받겠습니다."
