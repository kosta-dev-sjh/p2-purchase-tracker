# Naver OCR Parsing Strategy

작성일: 2026-04-27  
대상: `src/utils/ocrParsers.ts`, `src/utils/ocrAnalyzeImages.ts`, 향후 네이버 OCR 샘플 수집/검증 루프  
작성 목적: 네이버 주문내역 캡쳐의 1차 파싱 규칙과 AI 보정 계획, 그리고 UI 변경 대응 전략을 한 문서에 고정

---

## 1. 현재 상태 요약

현재 `parseNaverOrderText` 는 네이버 화면 구조를 거의 반영하지 못한 초기 스텁 수준이다.

- 상품명 후보를 `>` 로 끝나는 라인에서만 찾음
- 날짜/가격 후보를 `202x` 가 들어간 라인에서만 찾음
- 같은 결제 묶음인지, 접힌 주문인지, 펼친 주문인지 구분하지 않음
- 네이버의 "티켓처럼 점선으로 묶인 섹션" 개념을 전혀 사용하지 않음
- 결과적으로 `라인 개수 맞추기` 수준이지 `주문 구조 파싱` 수준은 아님

즉, 네이버 OCR은 쿠팡처럼 regex를 누적하기 전에 먼저 **레이아웃 관찰에 맞는 중간 표현(section/block)** 을 세워야 한다.

---

## 2. 네이버 화면에 대해 현재까지 합의된 관찰

사용자 관찰 기준으로 네이버 주문내역은 크게 두 모드가 있다.

### A. 접힌 주문

예상 구조:

```text
(상품명1) 포함 총 n건
총 n원
주문일자
주문 펼쳐보기 >
```

특징:

- 실제 상품은 여러 개인데 화면에는 대표 상품명 1개만 보임
- `포함 총 n건` 이 핵심 힌트
- `총 n원` 은 상품 1개의 금액이 아니라 **결제 섹션 전체 합계**
- 이 상태에서는 상품 2..n의 이름/개별 금액이 화면에 없음
- 따라서 1차 파서는 "개별 상품 fully recovered" 가 아니라 "접힌 묶음 draft" 로 저장해야 함

### B. 펼친 주문

예상 구조:

```text
(상품명1)
금액
(상품명2)
금액
(상품명3)
금액
...
주문일자
```

특징:

- 같은 결제에 속한 상품들이 한 섹션 안에 연속으로 존재
- 쿠팡보다 네이버 쪽이 "같이 결제한 상품 묶음" 경계가 더 분명할 가능성이 높음
- 주문일자가 상품마다 찍혀 보여도, 실제 저장 단위는 개별 상품이 아니라 **같은 섹션에 묶인 주문 그룹**

### C. 섹션 경계

중요한 관찰:

- 네이버는 UI적으로 티켓/점선 구분처럼 **한 결제 단위 섹션이 보이는 편**
- 즉 OCR 관점에서는 "상품 라인 추출"보다 먼저 "섹션을 어디서 끊을지"가 더 중요할 수 있음
- 1차 파서의 핵심 단위는 line이 아니라 `section`

---

## 3. 파싱 원칙

네이버는 쿠팡과 같은 방식으로 가면 안 된다.

### 원칙 1. 쿠팡처럼 상품 카드 단위 regex를 깊게 복제하지 않는다

- 쿠팡은 이미 parser-heavy 레거시 레인이다
- 네이버는 신규 플랫폼이므로 **얕은 1차 파서 + 이른 AI 보정**이 맞다
- 다만 "얕다"는 말이 "대충 라인만 매칭"을 뜻하진 않는다
- 최소한 `섹션 분리 -> 접힘/펼침 판정 -> draft 구조화` 는 해야 한다

### 원칙 2. 1차 파서는 완성본이 아니라 편집 가능한 초안만 만든다

1차 파서의 목표:

- 어떤 상품/합계/날짜가 어떤 섹션에 속하는지 최대한 안정적으로 나누기
- 펼친 주문은 가능한 한 개별 상품까지 초안 생성
- 접힌 주문은 "대표 상품 + 총액 + itemCount + folded=true" 형태로 남기기

1차 파서가 책임지지 않아도 되는 것:

- 접힌 주문의 숨겨진 상품 2..n의 이름 추정
- OCR로 뭉개진 브랜드명 정교 복원
- UI가 조금 비틀린 극단 케이스의 미세한 문구 복구

### 원칙 3. 네이버의 저장 단위는 "상품 1개"가 아니라 "같은 결제 섹션"

- 주문일자가 각 상품마다 반복 노출되어도 같은 섹션이면 같은 주문으로 본다
- 따라서 최종 `OcrOrder` 는 `section -> order`, `section 내부 상품들 -> products[]` 로 가는 것이 자연스럽다
- 접힌 주문도 1개 `OcrOrder` 로 저장하되, 내부 `products[]` 는 불완전 초안일 수 있다

## 4. 제안하는 중간 표현

네이버 파서를 바로 `PurchaseOCRResult[]` 로 만들기보다, 내부적으로는 아래 중간 구조를 먼저 만드는 편이 낫다.

```ts
interface NaverSectionDraft {
  rawLines: string[];
  date: string | null;
  statusText: string | null;
  isFolded: boolean;
  itemCountHint: number | null;
  sectionTotal: number | null;
  previewName: string | null;
  products: Array<{
    name: string | null;
    price: number | null;
  }>;
}
```

이 구조가 필요한 이유:

- 접힘/펼침 분기를 뒤로 미룰 수 있음
- AI에 보낼 때도 "이 섹션은 접힌 주문, 총 4건, 현재 보이는 건 1건뿐" 같은 강한 힌트를 줄 수 있음
- UI 수정 시에도 parser 전체를 뜯지 않고 section builder만 보완하면 됨

---

## 5. 1차 파서 설계

### Step 1. line 정규화

쿠팡 때와 같은 기본 전처리는 그대로 활용한다.

- 불필요한 leading pipe, OCR 장식 문자 제거
- 연속 공백 축소
- `원`, `총`, `건`, `주문`, `배송`, `취소`, `환불` 같은 핵심 토큰은 최대한 보존
- 날짜 패턴과 금액 패턴은 원형을 해치지 않게 유지

### Step 2. section 후보 분리

네이버는 먼저 "어디까지가 같은 주문 섹션인지" 나눠야 한다.

1차 구현 후보:

- 날짜 라인 기준으로 새 섹션 시작
- `주문 펼쳐보기`, `주문상세`, `구매확정`, `배송완료`, `환불완료`, `취소완료` 등 상태/액션 라인을 경계 힌트로 사용
- 빈 줄, 반복 divider OCR 패턴, 점선/티켓 경계에서 자주 나오는 문자 노이즈를 후보 경계로 사용

주의:

- 날짜가 각 상품마다 반복될 수 있으므로 "날짜 등장 = 무조건 새 order" 로 처리하면 안 됨
- 경계는 단일 힌트가 아니라 "날짜 + 상태 + 총액/상품 묶음 패턴" 조합으로 판단해야 한다

### Step 3. 접힘/펼침 판정

다음 중 하나라도 보이면 folded 후보로 분류:

- `포함 총 n건`
- `총 n건`
- `주문 펼쳐보기`
- 대표 상품명 1개 + 총액 1개만 있고 개별 가격 라인이 부족한 경우

펼침 후보 신호:

- 같은 섹션에서 상품명/가격 페어가 2쌍 이상 안정적으로 반복
- `총 n건`, `펼쳐보기` 없이 상품 라인이 연속

### Step 4. 섹션별 draft 생성

#### 4-1. folded section

생성 규칙:

- `previewName`: 대표 상품명
- `itemCountHint`: `포함 총 n건` 에서 추출
- `sectionTotal`: `총 n원` 에서 추출
- `products`: 일단 대표 상품 1개만 넣되, 금액은 개별 상품 금액이 아니라면 null 유지
- 별도 메타를 남겨 "이 섹션은 접혀 있음"을 caller가 알 수 있게 함

이 상태에서 중요한 건 정직함이다.

- 보이지 않는 상품을 추정해서 2..n개를 억지 생성하지 않는다
- 총액을 대표 상품 가격으로 오인해 넣지 않는다

#### 4-2. expanded section

생성 규칙:

- 섹션 내부에서 상품명/금액 페어를 위에서 아래 순서로 추출
- 같은 섹션에 속한 상품은 하나의 order로 합친다
- 각 상품마다 같은 날짜가 반복되더라도 order level date는 섹션 공통 날짜 1개로 정규화

### Step 5. `PurchaseOCRResult[]` 로 변환

현재 파이프라인을 크게 깨지 않으려면 최종적으로는 기존 타입으로 내려와야 한다.

권장 방식:

- expanded section: 상품마다 `PurchaseOCRResult` 생성, `rawText`/`statusText`/`date` 공유
- folded section: 대표 상품 1개만 `PurchaseOCRResult` 로 생성하되
  - `price` 는 대표 상품 금액이 확실하지 않으면 null
  - 혹은 `sectionTotal` 을 임시 price로 싣지 말고 별도 route를 고민

중요:

현재 타입만으로는 folded order의 "총합"과 "대표 상품"을 동시에 정직하게 담기 어렵다.  
따라서 네이버 도입을 본격화하면 `PurchaseOCRResult` 또는 `OcrOrder` 생성 직전 단계에 메타 확장이 필요할 수 있다.

## 6. folded order 처리 전략

이 부분이 네이버에서 가장 중요하다.

### 최소 안전 전략

접힌 주문은 아래처럼 다룬다.

- order 하나 생성
- 대표 상품 1개만 draft로 보여줌
- order level에 `folded`, `itemCountHint`, `sectionTotal`, `needsExpansionReview` 같은 메타를 남김
- UI에서 "이 주문은 접힌 상태로 추출되어 일부 상품이 숨겨져 있을 수 있음" 배지/안내를 붙임

장점:

- 거짓 정보를 만들지 않음
- 사용자에게 누락 위험을 명확히 알릴 수 있음

단점:

- 상품 단위 분석 정밀도는 낮아짐

### AI 보정 전략

접힌 주문을 AI로 보정할 때는 목표를 분리해야 한다.

가능한 목표:

1. 현재 화면에 실제로 보이는 텍스트만 정리
2. 접힌 주문임을 인식하고 "숨겨진 상품 있음"으로 표시
3. 이미지 상 다른 단서가 있으면 item count / total 정규화

하면 안 되는 목표:

- 이미지에 없는 상품 2..n의 이름을 환각으로 생성

즉 AI도 접힌 주문에서는 "복원"보다 "정직한 구조화" 쪽으로 써야 한다.

## 7. AI 보정 계획

## 7-1. "AI에게 이미지 구조를 알려주고 있는가?"

현재 구조상 `aiOcrFallback.ts` 는 플랫폼 힌트만 넘긴다.

- `platform: "coupang" | "naver"`
- `rawText`
- 전체 products
- bad card id 목록
- 원본 image

즉 지금도 완전히 blind는 아니지만, **구조적 힌트는 매우 약한 편** 이다.  
"이건 네이버 주문내역이고 접힌 주문이 있을 수 있으며, 총 n건/주문 펼쳐보기/섹션 경계를 봐라" 같은 지시는 아직 약하다.

### 개선 방향

AI 프롬프트 입력에 아래 정보를 추가한다.

- 플랫폼: 네이버
- 예상 UI 패턴:
  - 접힌 주문: `대표상품 포함 총 n건 + 총액 + 주문 펼쳐보기`
  - 펼친 주문: 같은 섹션 안에 상품명/가격이 반복
- 절대 금지:
  - 이미지에 보이지 않는 상품명을 추정 생성하지 말 것
- 요구 출력:
  - `folded / expanded`
  - `sectionCount`
  - `itemCountHint`
  - `sectionTotal`
  - `visibleProducts[]`
  - confidence / uncertainty notes

### 권장 역할 분담

- 1차 파서: section 분리, 접힘 감지, 날짜/상태/총액/대표상품 초안
- AI: 흐릿한 상품명 정리, OCR 파손 복구, folded 여부 검증, 섹션 오분리 교정

AI가 주도권을 가져야 하는 영역:

- OCR로 글자가 심하게 깨진 대표 상품명
- 섹션 경계가 애매한 이미지
- status/date/summary line이 뒤섞인 케이스

AI에게 맡기면 안 되는 영역:

- 안 보이는 상품 구성 추정
- total만 보고 개별 price 분배

## 8. "학습"을 어떻게 이해할지

여기서 말하는 학습은 두 가지로 나눠야 한다.

### A. 모델 재학습

현실적으로 지금 프로젝트 범위에서는 비추천이다.

- 비용 대비 효과가 낮다
- 데이터셋 관리, 라벨링, 재학습 파이프라인 운영이 무겁다
- 프론트엔드 앱 중심 구조와 맞지 않는다

### B. 샘플 기반 반복 개선 루프

이건 강하게 추천한다.

권장 루프:

1. 네이버 샘플 이미지를 모은다
2. 원본 이미지별 ground truth를 수작업으로 기록한다
3. 1차 파서 결과를 dump한다
4. AI 보정 결과를 dump한다
5. 둘을 비교해 failure taxonomy를 만든다
6. section 분리 규칙과 folded handling만 보강한다

즉 "학습"은 모델 파인튜닝보다 **harness 기반 반복 개선** 으로 가는 게 맞다.

이 방식이 좋은 이유:

- Claude가 1차 규칙/분류 기준을 잡기 좋다
- Codex가 회귀 방지, 구조 정리, 문서화, 파서 보완을 하기 좋다
- 이전 쿠팡 작업 방식과도 잘 맞다

---

## 9. Claude ↔ Codex 역할 분담 제안

사용자가 말한 협업 방식에 맞추면 아래가 자연스럽다.

### Claude에게 먼저 맡기기 좋은 일

- 네이버 샘플 이미지 관찰 메모 작성
- 이미지별 section 경계 수작업 라벨링
- folded/expanded 케이스 taxonomy 만들기
- 대표 실패 유형 정리
- "이 경우엔 어떤 draft가 정직한가" 초안 제안

### Codex가 이어받아 보완하기 좋은 일

- taxonomy를 코드 구조로 번역
- parser helper / section builder 설계
- ground-truth harness 정리
- 회귀 검증 루프와 문서 보강
- AI fallback 입력/출력 스키마 정리

즉 네이버 작업은 "Claude가 관찰과 taxonomy를 만들고, Codex가 구조화와 회귀 방지를 단단하게 하는" 방식이 잘 맞다.

## 10. UI가 바뀔 때의 대응 전략

이건 별도 운영 전략이 꼭 필요하다.

### 문제 정의

쇼핑몰 UI는 완전히 갈아엎지 않더라도 아래처럼 자주 변한다.

- 버튼 문구 변경
- summary line 위치 변경
- 접힘 CTA 문구 변경
- 날짜/상태/총액 배치 순서 변경
- divider/spacing 변경

이때 매번 사람이 눈으로 찾아 파서를 고치는 방식만으로는 유지비가 커진다.

### 권장 대응: 3중 방어선

#### 1. parser를 텍스트 literal이 아니라 구조 신호 중심으로 설계

예:

- `주문 펼쳐보기` 한 문구만 믿지 않음
- `총 n건`, 대표상품 1개, 총액 1개, 개별 가격 부족이라는 조합도 folded 신호로 사용
- 날짜 line과 금액 line의 상대 배치를 함께 봄

즉 UI 문구가 조금 바뀌어도 섹션 의미가 유지되면 견딜 수 있게 한다.

#### 2. drift 감지용 샘플 회귀 하니스 운영

권장 자산:

- `naver` 샘플 이미지 폴더
- raw OCR cache
- ground truth JSON
- parser output JSON
- AI output JSON
- 비교 리포트

새 샘플이 들어올 때 자동으로 아래를 계산한다.

- section 개수 일치율
- folded 감지 정확도
- itemCountHint 일치율
- sectionTotal 일치율
- visible product clean rate

#### 3. AI 기반 drift sentinel

이건 충분히 가능하다.

아이디어:

- 새 이미지가 들어오면 1차 파서 결과와 AI 결과를 비교
- 아래 조건이면 "UI drift candidate" 로 플래그

예시 조건:

- parser는 section 1개인데 AI는 section 3개로 봄
- parser는 folded 아님인데 AI는 folded 확신이 높음
- parser가 상품 0개로 봤는데 AI는 대표상품/총액을 안정적으로 읽음
- 특정 문구가 갑자기 여러 샘플에서 반복 등장

이렇게 되면 운영 로그나 dev 도구에서:

- `possible_ui_drift: true`
- `new_anchor_texts: ["주문 모두보기", "총 4개 상품"]`
- `samples_to_review: [...]`

형태로 개발자에게 알려줄 수 있다.

중요:

AI가 직접 parser를 수정하게 하기보다, **"드리프트 후보를 감지해 알려주는 역할"** 을 맡기는 편이 안전하다.

---

## 11. 추천 운영안

가장 현실적인 운영안은 아래다.

### 단기

- 네이버 샘플 수집
- folded / expanded / ambiguous 3분류로 라벨링
- 문서 기준으로 1차 section draft 파서 설계
- 접힌 주문은 정직하게 "숨김 상품 있음" 메타로 남김

### 중기

- 네이버 전용 harness 추가
- AI fallback 프롬프트에 "접힌 주문/섹션 경계" 지식 추가
- parser vs AI 차이가 큰 케이스를 자동 로그화

### 장기

- drift sentinel 구축
- UI 변경 후보 샘플을 주기적으로 리뷰
- 네이버 외 다른 플랫폼도 `section-first parser` 템플릿으로 확장

## 12. 바로 실행할 작업 제안

네이버 데이터를 수집한 뒤에는 아래 순서가 가장 효율적이다.

1. 샘플 이미지를 `folded`, `expanded`, `ambiguous` 로 나눈다
2. 이미지별로 "실제 order section 수 / 대표상품 / 총액 / 보이는 상품 수 / 주문일자" ground truth를 적는다
3. Claude가 샘플 관찰 문서와 failure taxonomy를 만든다
4. Codex가 `parseNaverOrderText` 를 section-first 구조로 리팩터링한다
5. AI fallback 입력 스키마에 folded/section 힌트를 추가한다
6. harness를 돌려 parser-only / parser+AI 결과를 비교한다

## 결론

네이버 OCR의 핵심은 "상품명 한 줄을 잘 읽는 것"이 아니다.  
핵심은 **같은 결제에 속한 섹션을 안정적으로 묶고, 접힌 주문을 정직하게 표현하는 것** 이다.

따라서 방향은 아래가 맞다.

- 1차 파서는 `section-first`
- 접힌 주문은 `숨겨진 상품 있음` 메타를 유지
- AI는 `보이는 구조 정리`와 `섹션 보정`에 사용
- UI 변경 대응은 사람이 매번 감으로 고치지 말고 `drift sentinel + harness` 로 운영

이 문서를 네이버 OCR 1차 구현의 기준 문서로 삼는다.
