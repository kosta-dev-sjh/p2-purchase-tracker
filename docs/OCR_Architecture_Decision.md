# OCR 파이프라인 아키텍처 결정 기록

**확정일**: 2026-04-24
**참여자**: Claude (세션 주 담당), Codex (교차 검토)
**대상 코드**: `src/utils/ocrAnalyzeImages.ts`, `src/utils/ocrParsers.ts`,
`src/utils/ocrQuality.ts`, `src/utils/aiOcrFallback.ts`, `src/utils/aiService.ts`

이 문서는 SpendTrack OCR 파이프라인이 **Tesseract + regex 파서 1차 + Gemini Flash
Vision 2차** 하이브리드를 유지하기로 결정한 배경과, 앞으로의 투자 방향을 남기는
기록입니다. 기획 문서는 아니고 엔지니어링 의사결정 기록(ADR 성격).

---

## 배경 — 왜 이 논의가 시작됐나

2026-04 세션 중 사용자의 실사용 체감 지적 3가지:

1. "샘플 5장 올렸는데 3번 이미지 빼고 다 AI 로 재분석된 것 같다. 1차 필터가 거의
   의미 없어 보인다."
2. "Gemini Flash 비용이 이렇게 저렴하면 Tesseract regex 튜닝하느라 시간 쓴 게
   무의미하지 않은가? AI-only 가 낫지 않나?"
3. "AI 보정 로딩 창이 '전체 이미지 수' 기준으로 진행률을 표시해, 실제로 건너뛴 이미지가
   있어도 모르게 된다."

(1), (3) 은 코드 버그(rule #4 과민 + progress 인디케이터 오표기) 로 확인 후 수정.
(2) 는 아키텍처 차원의 질문이라 두 AI 교차 검토로 결론 도출.

---

## 결론 요약

**하이브리드 유지. 단, Tesseract 투자 깊이는 플랫폼별로 차등.**

| 플랫폼 | 1차 파서 투자 수준 | AI 의존도 |
|---|---|---|
| 쿠팡 | parser-heavy (현재 23/23 harness). **동결 — 회귀 대응만** | 낮음 (bad 있을 때만) |
| 네이버쇼핑 (신규) | **얕은 1차 파서** — 주문/날짜/상태/상품명·가격의 대략적 추출만 | 높음 (세부 복구는 AI) |
| 테무 (신규) | 동상 | 동상 |
| 카드사 앱 등 (미래) | 동상 | 동상 |

"신규 플랫폼은 AI-first" 라는 표현은 **"AI-only" 로 오독되지 않도록** 주의. 편집
화면에 초안이 없으면 UX 치명적이라 1차 파서는 반드시 필요. 단 쿠팡처럼 규칙 복제를
깊게 할 필요는 없고, **"편집 가능한 구조화 초안"** 수준까지만.

## 각 대안의 득실 비교

| 축 | Tesseract-only | 하이브리드 (현재) | AI-only |
|---|---|---|---|
| 23장 harness 통과 | 23/23 ✓ | 23/23 ✓ | 미측정 (추정 더 높음) |
| 1장당 지연 | 2~5s (병렬) | 2~5s + 필요시 3~8s | 3~8s × N장 순차 |
| 5장 업로드 체감 | 10~15s | 10~20s | 20~40s |
| 월 30장/user 비용 | $0 | ~$0.002 (35% 호출) | ~$0.006 (100% 호출) |
| 10K user 월 비용 | $0 | ~$20 | ~$60 |
| 오프라인 작동 | ✓ (품질 저하) | ✓ (AI 만 실패) | ✗ |
| 개인정보 노출 | 로컬만 | 일부 외부 전송 | 모든 이미지 외부 전송 |
| **API 키 노출** | 해당 없음 | **번들 포함** (개발용만 가능) | **번들 포함** (배포 불가) |
| Rate limit 영향 | 없음 | 저 | 직격 (10 RPM 한도) |
| 코드 복잡도 | 파서 ~1000줄 | 파서 + AI 글루 ~1300줄 | AI 경로만 ~200줄 |
| 확장(신규 플랫폼) | 매번 parser 작성 | 얕은 파서 + AI 재사용 | 프롬프트만 수정 |

## Codex 가 지적한 결정적 맹점 — 배포 구조

Claude 의 초안은 **"비용만 고려"** 에 편향돼 있었음. Codex 가 지적한 핵심:

- `VITE_GEMINI_API_KEY` 가 프론트엔드 번들에 박히는 현재 구조는 **실 배포 부적합**.
  개발용으로만 통용되는 임시 설계.
- Gemini 2.5 Flash free tier: **10 RPM / 250 RPD** (2026-04 기준). 동시 사용자 10명이
  한꺼번에 업로드하면 free tier 초과. Paid tier 전환해도 rate limit 이 완화될 뿐
  사라지지 않음.
- 실 배포 전 **서버 프록시** (Vercel Functions / Cloudflare Workers / Next.js Route)
  로 키 은닉 + rate limit + 호출 로깅이 선행되어야 함.

이 맹점이 해결되기 전까지는 "AI 의존도를 지금보다 높이는 결정" 은 모두 유보.

## 의사결정 트리

### 지금 (유저 ~10명 내, 하루 업로드 ~20장)

- 하이브리드 유지.
- 쿠팡 파서는 **회귀 대응만**. 정확도 추가 상승 목적의 regex 추가 금지.
- 신규 플랫폼(네이버·테무) 추가 시 **얕은 1차 파서** + 이른 AI 보정.
- AI 변경 실효율 로깅 추가해 게이트 과민/둔감 판단용 수치 축적.

### 유저 100~1000명 구간

- **서버 프록시로 마이그레이션 선행**. 키 은닉 + per-user/IP rate limit + 호출 로깅.
- 이후 "Tesseract 프리뷰 → 백그라운드 AI 교정" UX 패턴 실험 가능.
- AI 변경 실효율 지표로 게이트 임계값 미세 조정.

### 유저 1000명+

- 서버 프록시가 전제된 상태에서 AI-always 모드 검토 가능.
- 다만 쿠팡은 로컬 Tesseract 가 여전히 속도 우위가 있어 완전 AI-only 전환은 여전히
  비권장. 쿠팡 레거시 레인 + 신규 AI 레인 **이중 레인** 유지가 현실적.
- Gemini 외 대안(Claude Haiku 등) fallback 레이어 검토 — rate limit 초과 시 자동
  전환.

## 합의된 용어 정리

- **bad 카드**: `classifyOcrCardQuality` 가 tier `bad` 로 분류한 카드. AI 자동 호출 대상.
- **1차 필터(= 게이트)**: "bad 카드가 있는 이미지만 AI 호출" 하는 현재 로직.
- **AI 변경 실효율**: AI 호출 후 실제로 `aiApplied: true` 가 찍힌 카드/이미지 비율.
- **얕은 파서**: 주문 단위 분리·날짜·상태·상품명·가격의 **대략적** 추출까지만 책임지는
  파서. 세밀 복원은 AI 담당.
- **parser-heavy legacy lane**: 쿠팡처럼 이미 regex 자산이 누적된 플랫폼 경로.
  회귀 대응만 하고 추가 투자 금지.

## 미해결 / 향후 검토

1. **AI 변경 실효율의 "합리적" 임계치** — 현재는 측정 안 되므로 기준 설정 불가. 우선
   로깅 시작 후 1~2주 데이터 축적 후 재검토.
2. **쿠팡 외 플랫폼 샘플 수집** — 네이버쇼핑·테무 캡쳐 샘플이 부족해 ground-truth
   harness 를 쿠팡 수준으로 만들 수 없음. 우선 얕은 파서로 초안 만들고 AI 결과의
   수동 편집률로 품질 측정.
3. **Claude 2차 fallback 유용성** — Gemini 로 부족한 1~2% 극단 케이스에 Claude
   Sonnet 을 붙이는 게 ROI 있는지 미검증.
