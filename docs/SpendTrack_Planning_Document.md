## 1. 프로젝트 개요

### 1-1. 프로젝트명
SpendTrack

### 1-2. 서비스 한 줄 정의
SpendTrack는 쇼핑 주문내역 캡처를 OCR로 분석해 구매 상품 정보를 복원하고, 이를 수입·지출 흐름 안에서 기록, 수정, 조회, 분석할 수 있도록 돕는 가계부형 소비관리 웹앱입니다.

### 1-3. 서비스 정체성
SpendTrack는 단순한 OCR 도구도 아니고 종합 금융관리 서비스도 아닙니다.  
이 서비스는 `쇼핑 주문내역 OCR`을 차별점으로 삼되, 결과적으로는 `생활형 소비관리`까지 이어지는 구조를 목표로 합니다.

| 항목 | 정의 |
| --- | --- |
| 서비스 포지션 | 쇼핑 주문내역 OCR 기반 가계부형 소비관리 서비스 |
| 차별점 | 결제내역이 아닌 실제 구매 상품 맥락 복원 |
| 핵심 사용자 가치 | 기록 장벽 감소, 구매 추적 가능, 소비 흐름 확인 |
| 프로젝트 성격 | 프론트엔드 중심 MVP 구현 |
| 팀 구성 | 2인 프론트엔드 세미프로젝트 |

### 1-4. 프로젝트 배경
기존 카드사, 은행 앱, 간편결제 서비스에서는 결제 내역에 보통 `쿠팡`, `네이버쇼핑`, `11번가` 같은 플랫폼명과 금액 정도만 표시됩니다.  
이 경우 사용자는 실제로 무엇을 구매했는지, 환불인지, 정기결제인지, 어떤 상품과 관련된 지출인지 추적하기 어렵습니다.

기존 가계부 앱 역시 수동 입력의 번거로움 때문에 지속 사용이 어렵고, 카드 내역만으로는 상세 맥락을 복원하기 어렵습니다.

원래 기획은 쇼핑몰 API 또는 크롤링을 통한 자동 통합을 지향했지만, 세미프로젝트 범위와 보안/구현 현실성을 고려했을 때 직접 연동 중심 접근은 무겁다고 판단했습니다.  
따라서 SpendTrack는 이를 `OCR 기반 반자동 입력 + 사용자 수정 방식`으로 현실화하는 방향을 선택했습니다.

### 1-5. 공식 API 부재와 OCR 선택의 타당성

SpendTrack의 차별점인 `실제 구매 상품 맥락 복원`을 위해서는, 이상적으로는 쇼핑몰이 사용자 본인의 주문내역 데이터를 직접 외부에 제공해 주는 공식 API를 호출하는 것이 가장 정확합니다.  
그러나 2026-04 기준 국내 환경에서는 이 경로가 구조적으로 닫혀 있으며, 이것이 SpendTrack가 OCR 기반 접근을 채택한 근본적인 이유입니다.

#### 1-5-1. 국내 주요 플랫폼의 개인 구매자 주문내역 API 제공 현황

| 플랫폼 | 개인 주문내역 API | 비고 |
| --- | --- | --- |
| 쿠팡 | 없음 | 쿠팡 파트너스(Seller) API만 존재하며 판매자 전용 |
| 네이버쇼핑 | 없음 | 공개된 것은 상품 검색 API뿐 |
| 네이버페이 | 없음 | 가맹점 결제 연동용 API만 공개 |
| 무신사 | 없음 | 공식 OpenAPI 자체가 존재하지 않음 |
| 11번가 / G마켓 / 옥션 | 없음 | 동일 |
| 카카오페이 / 토스페이 | 없음 | 가맹점 결제 API 중심 |

즉 국내 어느 쇼핑몰도 `이 사용자의 주문내역을 외부 서비스가 읽어가세요`라는 형태의 공식 API를 제공하지 않습니다.  
이는 서비스 규모의 문제가 아니라, 개인정보·결제정보 보호 관점에서 구조적으로 차단된 영역이기 때문입니다.

#### 1-5-2. 우회 경로 검토

| 경로 | 접근 가능성 | 한계 |
| --- | --- | --- |
| CODEF / 오픈뱅킹 (카드·은행 거래내역 스크래핑) | 일부 가능 | 결제처와 금액만 확인 가능하고, 실제 구매 상품 상세는 확인할 수 없음 |
| MyData 사업자 API | 불가능 | 신용정보법상 지정된 마이데이터 사업자만 접근 가능 |
| 이메일 주문확인 메일 파싱 | 제한적 가능 | OAuth 심사, 플랫폼별 메일 포맷 차이, 유지보수 비용 누적 |
| 크롤링 / 헤드리스 브라우저 | 기술적 가능 | 이용약관 위반, 로그인·2FA·캡차 대응, 세션 유지 부담 |

특히 CODEF나 오픈뱅킹 같은 경로는 `쿠팡 결제 139,000원`과 같은 결제내역 레벨의 데이터만 제공합니다.  
SpendTrack이 해결하려는 `실제로 어떤 상품을 샀는지까지 복원`은 이 경로로 재현할 수 없습니다.

#### 1-5-3. 결론: OCR 기반 접근은 타협이 아니라 현재 시장 구조상 가장 현실적인 경로

이 구조적 공백을 메울 수 있는 현실적인 방법은, 사용자가 이미 가지고 있는 주문내역 화면 캡처나 영수증 이미지를 OCR로 읽어 구매 맥락을 복원하는 것입니다.  
OCR은 인식 정확도가 완벽하지 않지만, SpendTrack은 이를 다음 두 단계 구조로 보완합니다.

1. OCR이 구매 상품 텍스트를 1차 추출해 수동 입력 부담을 줄입니다.
2. 사용자가 OcrEdit 화면에서 직접 결과를 수정해 데이터 정확성을 확보합니다.

따라서 SpendTrack이 OCR 기반 반자동 입력을 핵심 기술로 채택한 것은 단순히 구현 난이도를 낮추기 위한 타협이 아니라, 현재 국내 시장 구조상 사용자의 실제 구매 맥락을 합법적으로 복원할 수 있는 사실상 유일한 경로이기 때문입니다.

#### 1-5-4. 현재 구현 기준 OCR 엔진 전략

OCR 자체를 제품 차별점으로 유지하면서도 비용과 정확도를 함께 관리하기 위해, SpendTrack는 다음 구현 단계에서 아래와 같은 **하이브리드 OCR 전략**을 기준으로 삼습니다.

1. 기본 추출은 `Tesseract.js`로 수행합니다.
2. 추출 결과를 구조화한 뒤 validator로 품질을 검사합니다.
3. 필수 필드 누락 또는 저신뢰 케이스에 한해서만 Vision LLM fallback을 호출합니다.
4. 최종 결과는 OcrEdit 화면에서 사용자가 수정·확정합니다.

즉 SpendTrack의 핵심은 "AI가 모든 것을 자동으로 확정한다"가 아니라,  
`저비용 기본 OCR + 필요한 경우만 AI 보조 + 사용자 최종 수정` 구조를 통해 현실적인 정확도와 제어 가능성을 함께 확보하는 데 있습니다.

### 1-6. 실 데이터 반입 경로 (CSV + OCR 결합 구조)

공식 API 부재라는 구조적 제약 안에서도 SpendTrack은 사용자가 실제 본인 데이터를 반입할 수 있는 경로를 제공해야 합니다.  
이를 위해 본 MVP에서는 `결제내역 레벨의 벌크 반입`과 `상품 상세 레벨의 단건 반입`을 각각 다른 경로로 처리하고, 두 경로를 하나의 거래 단위에서 결합하는 구조를 채택합니다.

#### 1-6-1. 두 축 구조

| 축 | 반입 경로 | 제공 데이터 | 역할 |
| --- | --- | --- | --- |
| 결제내역 | 카드사 CSV 업로드 | 플랫폼(가맹점), 금액, 결제일 | 월 단위 벌크 입력으로 수동 입력 부담 제거 |
| 상품 맥락 | 주문 캡처 OCR | 상품명, 개별 가격, 구매 시점 상세 | 결제내역만으로는 부족한 상품 맥락 복원 |

두 축은 배타적인 관계가 아니라 **보완 관계**입니다. CSV는 한 번에 수십 건의 결제를 빠르게 채워 넣고, OCR은 개별 거래의 상세 맥락을 덧붙이는 역할을 담당합니다.

#### 1-6-1-A. 현재 플랫폼 지원 정책

입력 경로마다 지원 범위를 동일하게 두지 않고, 제품 역할에 따라 아래처럼 구분합니다.

- 쇼핑 OCR과 쇼핑 중심 수동 입력은 현재 `쿠팡`, `네이버쇼핑`, `무신사` 3개 플랫폼에 우선 최적화합니다.
- 카드내역 CSV/엑셀/PDF 반입은 가계부 보조 입력축으로 보고, 카드사 제한을 두지 않고 최대한 넓게 수용합니다.

즉 SpendTrack는 `쇼핑 맥락 복원` 영역에서는 명확한 플랫폼 최적화를 유지하고,  
`가계부 반입` 영역에서는 더 넓은 파일 호환성을 우선하는 이중 전략을 채택합니다.

#### 1-6-2. 결합 방식

CSV로 들어온 거래(상품 상세 없음)와 OCR로 추출한 거래(상품 상세 있음)는 `플랫폼 + 금액 + 날짜` 기준으로 자동 매칭 후보를 탐색합니다.  
매칭 후보가 있는 경우 사용자는 두 가지 선택지 중 하나를 고릅니다.

1. 기존 거래에 상품을 추가해 결제내역과 상품 맥락을 하나로 병합
2. 별도의 새 거래로 저장해 두 흐름을 분리 유지

MVP 단계에서는 **자동 병합을 강제하지 않고 사용자가 항상 확인 후 선택**하도록 설계합니다. 이는 잘못된 자동 매칭으로 인해 데이터 신뢰도가 깨지는 상황을 방지하기 위함입니다.

#### 1-6-3. 카드 API 연동과의 관계

현재 MVP에서 카드사/은행의 실 연동 API(CODEF, 오픈뱅킹, MyData 등)는 도입하지 않습니다.  
이유는 1-5-2절에서 정리한 것처럼 인증 구조 복잡성, 공동인증서 의존, 건당 과금, 개인 개발자 접근 제약 등 MVP 범위에서 감당하기 어려운 부담이 크기 때문입니다.

대신 사용자가 카드사 앱/웹에서 직접 내려받은 CSV를 업로드하는 방식은 다음 이점이 있습니다.

- 본인 데이터를 본인이 업로드하므로 인증·약관 이슈에서 자유롭습니다.
- 국내 대부분의 카드사가 월별 이용내역 CSV/Excel 다운로드를 지원합니다.
- 네트워크 연동이 없어 오프라인 시연에서도 동일하게 동작합니다.

따라서 CSV 업로드는 `카드 API 연동`의 대체재가 아니라, MVP 단계에서 실 데이터 반입을 가능하게 하는 **현실적 선행 단계**입니다.  
향후 실 금융 API 연동이 가능한 단계로 확장될 경우, CSV 업로드 경로는 자동 동기화 경로와 병존하며 `오프라인·수동 반입` 옵션으로 유지될 수 있습니다.

#### 1-6-4. 저장 구조

초기 MVP 단계에서는 브라우저 localStorage 기반의 임시 저장소로 UX를 먼저 검증했고,  
현재 구현 기준에서는 Firebase Authentication + Firestore 기반 구조로 전환하는 방향을 확정했습니다.

이 판단의 이유는 다음과 같습니다.

- SpendTrack는 사용자별 거래 데이터가 누적되는 구조이므로 `로그인/회원가입`, `사용자별 데이터 분리`, `원격 저장`이 필수입니다.
- 프론트엔드 중심 팀 기준으로 Firebase는 비교적 짧은 기간 안에 안정적으로 붙이기 좋은 managed backend입니다.
- 이 프로젝트의 핵심은 `가계부 도메인 + OCR/CSV 결합 경험`이지, 서버 인프라 자체를 직접 운영하는 경험이 아닙니다.

현재 기준 저장 계층은 다음처럼 정리합니다.

- 인증: Firebase Authentication
- 사용자별 거래/카테고리/프로필 저장: Firestore
- 외부 AI 호출 보호 계층: Firebase Functions

즉 localStorage는 기획 방향이 아니라 **초기 퍼블리싱 단계의 임시 저장 방식**이었고,  
이제 SpendTrack의 실구현 기준 아키텍처는 Firebase 중심 구조로 보는 것이 맞습니다.

#### 1-6-5. Firebase 중심 구조를 택한 이유

SpendTrack는 현재 프로젝트 규모와 팀 구성, 그리고 구현 우선순위를 고려할 때  
EC2/Docker 기반 인프라까지 함께 운영하기보다 Firebase 중심으로 정리하는 편이 더 적절합니다.

| 구분 | 역할 |
| --- | --- |
| Firebase Authentication | 로그인, 회원가입, 세션 관리 |
| Firestore | 사용자별 거래/카테고리/프로필 저장 |
| Firebase Functions | 외부 AI API 키 보호, AI 호출 프록시 |
| Firebase Hosting | 프런트엔드 SPA 배포 |

이 구조를 선택한 이유는 다음과 같습니다.

- 인증, 데이터 저장, AI 호출 보호 계층을 한 서비스군 안에서 통합 관리할 수 있습니다.
- 프런트엔드 중심 팀이 별도 서버 인프라를 관리하지 않고도 기능 완성에 집중할 수 있습니다.
- 프로젝트 규모상 EC2/Docker를 함께 도입하면 학습 요소는 늘지만, 실제 제품 완성 속도와 운영 단순성은 오히려 떨어질 수 있습니다.

따라서 현재 SpendTrack의 인프라 방향은 `Firebase 중심 단일 스택`으로 정리합니다.  
EC2/Docker는 본 프로젝트의 필수 범위가 아니며, 별도 학습 주제 또는 후속 확장 주제로 분리하는 것이 더 적절합니다.

## 2. 문제 정의

### 2-1. 핵심 문제

| 문제 | 설명 |
| --- | --- |
| 구매 맥락 부재 | 결제내역에는 플랫폼명과 금액만 보이고 실제 상품 정보는 보이지 않음 |
| 기록 분산 | 쇼핑몰별 주문내역이 흩어져 있어 통합 관리가 어려움 |
| 수동 입력 부담 | 가계부는 직접 기록해야 해서 귀찮고 지속 사용이 어려움 |
| 상태 추적 어려움 | 환불, 취소, 반품, 정기결제를 한눈에 파악하기 어려움 |
| 분석 부족 | 단순 금액 기록만으로는 소비 패턴을 이해하기 어려움 |

## 5. AI 적용 방향

### 5-1. AI 도입 필요성
강사 피드백 기준으로 AI 기능을 포함하면 프로젝트 완성도와 차별성이 높아질 수 있습니다.  
다만 팀 규모가 2인이고 핵심 기능도 많은 편이므로 AI는 `챗봇`이 아니라 `간단한 소비 인사이트 생성 기능 + OCR 보조 fallback` 정도로 제한하는 것이 적절합니다.

즉 SpendTrack에서 AI는 별도 대화형 기능이 아니라, 다음 두 역할로 제한합니다.

1. 저장된 거래 데이터를 읽고 짧은 분석 메시지를 출력하는 보조 기능
2. 기본 OCR이 불안정한 케이스에서만 구조화 추출을 보조하는 fallback 기능

### 5-2. AI 적용 결정

| 항목 | 결정 |
| --- | --- |
| 챗봇 도입 | 제외 |
| 대화형 AI | 제외 |
| 추천 엔진 | 제외 |
| 간단한 소비 인사이트 | 포함 |
| OCR fallback 보조 | 포함 |
| 구현 수준 | 규칙 기반 요약 데이터 + AI 문장화, 기본 OCR 실패 시 Vision fallback |

### 5-3. 적용 가능한 방식

#### 방식 A. 규칙 기반 인사이트
- 이번 달은 온라인 쇼핑 지출 비중이 높아요.
- 환불 건수가 지난달보다 2건 늘었어요.
- 정기결제로 추정되는 거래가 3건 있어요.

장점:
- 구현이 가장 단순함
- 결과 통제가 쉬움

한계:
- AI 느낌은 약함

#### 방식 B. 규칙 기반 요약 데이터 + AI 문장화
- 총 지출, 총 수입, 상위 플랫폼, 환불 건수, 정기결제 추정 건수 등을 먼저 구조화
- 그 값을 바탕으로 AI가 1~2문장 메시지를 생성

장점:
- 발표 시 AI 활용 설명이 쉬움
- 분석 책임은 서비스가 유지하고, 문장 생성만 AI가 맡음

#### 방식 C. 기본 OCR + Vision fallback
- 1차 추출은 Tesseract.js로 수행
- 주문일자, 총 금액, 상품명 등 필수 필드 누락 여부를 validator로 검사
- 구조화 결과가 기준을 통과하지 못할 때만 Vision LLM을 호출
- 결과는 사용자가 OcrEdit에서 최종 수정

장점:
- OCR을 프로젝트 차별점으로 유지할 수 있음
- 모든 이미지에 AI 비용이 발생하지 않아 비용 통제가 쉬움
- "기본 분석 후 어려운 건 AI에 맡긴다"는 제품 서사를 UX로 드러낼 수 있음

#### 방식 D. 파일 반입 파싱 + AI fallback
- 카드사 CSV/엑셀/PDF 파일은 우선 규칙 기반 파서로 처리
- 플랫폼/카드사별 헤더, 날짜, 금액 형식은 팀이 1차적으로 직접 매핑
- 규칙 파서가 구조를 잡지 못하는 경우에만 AI에게 재해석을 요청
- AI 결과도 곧바로 확정하지 않고, 미리보기/검토 단계를 거쳐 저장

장점:
- 파일 반입의 설명 가능성과 예측 가능성을 유지할 수 있음
- 카드사 양식이 늘어날 때 모든 예외를 코드로만 처리해야 하는 부담을 줄일 수 있음
- API 승인 없이도 다양한 카드사 파일을 대응할 확장 경로를 확보할 수 있음

### 5-4. 최종 결정
이번 프로젝트에서는 아래 두 가지 AI 적용 방식을 최종 채택합니다.

1. `규칙 기반 요약 데이터 + AI 문장화`
2. `기본 OCR + Vision fallback`

파일 반입(CSV/엑셀/PDF) 영역은 별도 축으로 보되, 다음 구현 단계에서는 아래 원칙을 적용합니다.

- 1차 파싱은 팀이 직접 작성한 규칙 기반 파서가 담당
- AI는 규칙 파서 실패 케이스를 줄이기 위한 보조 fallback으로만 사용
- 즉 파일 반입의 기본 책임은 여전히 서비스 내부 파서가 갖고, AI는 예외 처리 범위를 넓히는 역할로 제한

### 5-5. 적용 방식
1. 프론트엔드에서 월별 거래 데이터를 요약합니다.
2. 총 지출, 총 수입, 상위 플랫폼, 환불 건수, 정기결제 추정 건수 등을 구조화합니다.
3. Firebase Cloud Functions를 통해 외부 LLM API를 호출합니다.
4. AI는 이 요약값을 바탕으로 1~2문장의 소비 인사이트를 생성합니다.
5. 생성된 메시지는 우선 Analysis의 상단 SummaryBanner에 노출합니다.
6. OCR 경로에서는 Tesseract.js 결과를 validator가 검사하고, 실패 시에만 Firebase Functions를 통해 Vision LLM을 호출합니다.
7. CSV/엑셀/PDF 경로에서는 규칙 기반 파서가 먼저 구조화를 시도하고, 실패 시에만 AI에게 파일 구조 재해석을 요청합니다.

### 5-6. AI 호출 타이밍 원칙

SpendTrack의 AI는 화면 진입마다 반복 호출하지 않고, **생성/저장 시점에만 실행**하는 것을 원칙으로 합니다.

- SummaryBanner 인사이트는 월별 거래 데이터가 실제로 갱신되었을 때만 새로 생성합니다.
- OCR fallback은 사용자가 분석을 시작한 이미지에 대해 1회 분석 흐름 안에서만 수행합니다.
- Analysis, Home, OcrEdit 화면 재진입만으로는 AI를 다시 호출하지 않습니다.

이 원칙은 비용 통제뿐 아니라, 사용자에게 동일한 데이터에 대해 매번 다른 문장이 나타나는 혼란을 줄이는 데도 중요합니다.

### 5-7. OCR 처리 상태 노출 원칙

OCR 분석은 처리 시간이 케이스별로 달라질 수 있으므로, SpendTrack는 다음 구현 단계에서 분석 과정을 사용자에게 단계형 상태로 보여주는 것을 권장합니다.

- 이미지 분석 중이에요...
- 인식 결과를 검토하고 있어요...
- AI에게 맡깁니다... (fallback 발생 시에만 노출)
- 분석 완료!

이 상태 노출은 단순 로딩 연출이 아니라,  
`기본 OCR → 검증 → 필요 시 AI fallback` 구조를 사용자 경험 차원에서 이해시키는 장치로 봅니다.

## 7. 기술 스택

### 7-1. 기획 기준 기술 스택

| 영역 | 기술 |
| --- | --- |
| Frontend | React, TypeScript, Vite |
| OCR | Tesseract.js |
| AI | Vision LLM fallback + LLM text generation |
| Styling | styled-components |
| Backend/DB | Firebase Authentication, Firebase Firestore |
| Serverless/API 보호 | Firebase Cloud Functions |
| 배포/운영 | Firebase Hosting |
| 협업/CI/CD | GitHub Actions |
| 디자인 작업 | Figma + MCP 기반 작업 |

기획 기준에서 인증/데이터/AI 보호/배포를 모두 Firebase 중심으로 정리한 이유는 다음과 같습니다.

- Firebase는 로그인, 사용자별 데이터 분리, 영속 저장을 담당하는 서비스 계층입니다.
- Firebase Functions는 외부 AI API 키를 브라우저에 노출하지 않고 호출하기 위한 보조 계층입니다.
- Firebase Hosting은 현재 규모의 SPA 배포에 필요한 라우팅/정적 배포 요건을 충분히 만족합니다.
- 따라서 현재 프로젝트에서는 별도 EC2/Docker 인프라를 두기보다, Firebase 안에서 한 번에 정리하는 쪽이 더 적절합니다.
